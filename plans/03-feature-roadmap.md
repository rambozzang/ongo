### 1.2 Publer 취약점 (사용자 리뷰 기반)

| 취약점 | 심각도 | onGo 공략 전략 |
|--------|--------|---------------|
| **분석이 표면적**, 경쟁자 벤치마킹 없음 | 높음 | AI 기반 심층 분석 + 경쟁 크리에이터 벤치마킹 |
| **팀 협업 기본적**, 클라이언트/에이전시 관리 없음 | 높음 | MCN/에이전시급 팀 워크플로우 |
| **대량 영상 업로드 불안정** (크래시) | 높음 | Tus Protocol 이어받기 + 안정적 벌크 업로드 |
| **미디어 라이브러리 빈약** (폴더/태깅 없음) | 중간 | 스마트 에셋 라이브러리 (AI 자동 태깅) |
| **화이트라벨 없음** | 중간 | 에이전시용 화이트라벨 지원 |
| **고객 지원 느림** (3일 응답) | 중간 | 24시간 내 응답 + AI 챗봇 자가 해결 |
| **모바일 앱 기능 부족** | 중간 | 모바일 웹 최적화 우선, Phase 3 네이티브 앱 |
| **제한적 외부 연동** | 중간 | API/웹훅 개방 + 주요 서비스 연동 |

### 1.3 onGo 경쟁 우위 (이미 보유)

1. **한국 시장 특화**: 네이버 클립 지원, 한국어 네이티브, 카카오 로그인
2. **AI 역량 압도적 우위**: 8종 AI 도구 vs Publer의 기본 AI Assist
3. **안정적 대용량 업로드**: Tus Protocol 기반 이어받기 업로드
4. **투명한 AI 과금**: 크레딧 기반 종량제 (사용한 만큼만 과금)
5. **영상 크리에이터 특화**: 텍스트 SNS가 아닌 영상 중심 설계

## 2. 전략 프레임워크

### 2.1 3단계 전략

```
Phase 1 (1-2개월): "따라잡기" - Publer 동등 수준 달성
    목표: 사용자가 Publer에서 넘어올 때 "없는 기능"이 없게
    핵심: 벌크 작업, 콘텐츠 재활용, 알림 시스템, 기본 팀 강화

Phase 2 (3-4개월): "넘어서기" - Publer가 못하는 것을 하기
    목표: "onGo에서만 가능한 것"을 만들어 전환 동기 부여
    핵심: AI 영상 리사이징, 경쟁자 벤치마킹, 에이전시 모드, 스마트 라이브러리

Phase 3 (5-6개월): "지배하기" - 한국 시장에서 대체 불가능한 위치
    목표: 네트워크 효과와 생태계 lock-in으로 시장 지배
    핵심: 크리에이터 마켓플레이스, 커머스 연동, API 생태계, 화이트라벨
```

### 2.2 기능 평가 기준

| 기준 | 가중치 | 설명 |
|------|--------|------|
| 비즈니스 임팩트 | 35% | 유료 전환, ARPU, 리텐션에 미치는 영향 |
| 경쟁 차별화 | 25% | Publer 대비 독자적 가치 |
| 구현 난이도 | 20% | 개발 리소스, 기술적 복잡도, 외부 의존성 |
| 사용자 요구 | 20% | 페르소나 기반 니즈 충족도 |

### 2.3 난이도/임팩트 표기 체계

- **구현 난이도**: S(간단, 1-2주) / M(보통, 2-4주) / L(어려움, 4-8주) / XL(매우 어려움, 8주+)
- **비즈니스 임팩트**: ★☆☆(낮음) / ★★☆(중간) / ★★★(높음)
- **우선순위**: P0(필수) / P1(높음) / P2(중간) / P3(낮음)

## 3. Phase 1: MVP 강화 (1-2개월)

> **목표**: Publer 동등 수준 달성. 기존 사용자 이탈 방지 및 Publer 사용자가 전환할 때 기능 부재로 인한 이탈이 없도록 한다.

### 3.1 벌크 작업 시스템

Publer의 CSV 일괄 스케줄링과 벌크 업로드를 능가하되, Publer의 치명적 약점인 "대량 영상 업로드 크래시"를 해결한다.

#### 3.1.1 벌크 영상 업로드

| 항목 | 내용 |
|------|------|
| **설명** | 최대 50개 영상을 한 번에 드래그&드롭으로 업로드. 각 영상에 개별 메타데이터 설정 가능 |
| **구현 난이도** | M (3주) |
| **비즈니스 임팩트** | ★★★ |
| **우선순위** | P0 |
| **예상 기간** | 3주 |

**상세 기능 명세**:

- Tus Protocol 기반 다중 파일 동시 업로드 (최대 5개 동시 전송)
- 개별 파일 프로그레스바 + 전체 진행률 표시
- 실패한 파일만 선택적 재시도 (Publer 대비 핵심 차별점)
- 업로드 큐 관리: 일시정지/재개/취소/우선순위 변경
- 백그라운드 업로드 지원 (브라우저 탭 이동해도 계속)

**기술 구현 방안**:

```
Frontend:
- 큐 매니저 (Vuex/Pinia store) + Web Worker로 업로드 프로세스 분리
- Intersection Observer로 보이는 파일만 프로그레스 렌더링 (성능 최적화)

Backend:
- Tus 서버 다중 세션 관리
- Spring Events로 업로드 완료 이벤트 비동기 처리
- Virtual Thread 활용 동시 업로드 핸들링
```

#### 3.1.2 CSV/엑셀 일괄 스케줄링

| 항목 | 내용 |
|------|------|
| **설명** | CSV 또는 엑셀 파일로 다수의 예약 게시를 한번에 등록. 템플릿 다운로드 제공 |
| **구현 난이도** | M (2주) |
| **비즈니스 임팩트** | ★★☆ |
| **우선순위** | P1 |
| **예상 기간** | 2주 |

**상세 기능 명세**:

- CSV 템플릿: `제목, 설명, 태그(;구분), 플랫폼(YT;TT;IG;NV), 예약일시, 공개설정`
- 엑셀(xlsx) 지원 (Apache POI)
- 업로드 전 데이터 검증 미리보기 (오류 행 하이라이트)
- 기존 업로드된 영상과 매칭 (파일명 기반)
- 최대 200건 일괄 등록

#### 3.1.3 벌크 액션 (일괄 편집/삭제/재업로드)

| 항목 | 내용 |
|------|------|
| **설명** | 영상 목록에서 체크박스 다중 선택 후 일괄 작업 수행 |
| **구현 난이도** | S (1주) |
| **비즈니스 임팩트** | ★★☆ |
| **우선순위** | P1 |
| **예상 기간** | 1주 |

**지원 액션**:

- 일괄 삭제 (확인 모달 필수)
- 일괄 재업로드 (실패한 플랫폼만 선택적)
- 일괄 태그 추가/변경
- 일괄 카테고리 변경
- 일괄 공개 설정 변경

### 3.2 콘텐츠 재활용 (Recycling) 시스템

Publer의 인기 기능인 콘텐츠 재활용을 한국 시장에 맞게 구현하되, AI 기반으로 고도화한다.

#### 3.2.1 기본 콘텐츠 재활용

| 항목 | 내용 |
|------|------|
| **설명** | 과거 게시 영상을 자동으로 큐에 추가하여 재게시. 주기/조건 설정 가능 |
| **구현 난이도** | M (3주) |
| **비즈니스 임팩트** | ★★☆ |
| **우선순위** | P1 |
| **예상 기간** | 3주 |

**상세 기능 명세**:

- 재활용 큐 생성: 영상 그룹 선택 + 재게시 주기(매주/격주/월) + 시간대 설정
- 재활용 규칙: "조회수 1만 이상인 영상만", "최근 3개월 내 게시한 영상", "특정 카테고리"
- 제목/설명 변형 옵션: 동일 / AI가 약간 변형(1크레딧) / 직접 수정
- 재활용 캘린더에 표시 (일반 예약과 구분, 점선 테두리)
- 재활용 중복 방지: 같은 영상을 같은 플랫폼에 7일 이내 재게시 경고

#### 3.2.2 에버그린 콘텐츠 태그

| 항목 | 내용 |
|------|------|
| **설명** | 시의성 없는 "상록수" 콘텐츠를 태깅하여 재활용 큐에 자동 추가 |
| **구현 난이도** | S (1주) |
| **비즈니스 임팩트** | ★☆☆ |
| **우선순위** | P2 |
| **예상 기간** | 1주 |

### 3.3 실시간 알림 시스템

현재 onGo의 알림 시스템을 실시간으로 강화하여, 크리에이터가 항상 최신 상태를 파악할 수 있도록 한다.

#### 3.3.1 통합 알림 센터

| 항목 | 내용 |
|------|------|
| **설명** | 모든 알림을 한 곳에서 관리. 실시간 WebSocket + 이메일/카카오톡 알림 |
| **구현 난이도** | M (3주) |
| **비즈니스 임팩트** | ★★★ |
| **우선순위** | P0 |
| **예상 기간** | 3주 |

**알림 유형**:

| 카테고리 | 알림 항목 | 실시간 | 이메일 | 카카오 |
|---------|---------|--------|--------|--------|
| 업로드 | 업로드 성공/실패 | O | O | O |
| 업로드 | 플랫폼 심사 통과/반려 | O | O | O |
| 예약 | 예약 게시 30분 전 | O | O | - |
| 예약 | 예약 게시 완료/실패 | O | O | O |
| 채널 | 토큰 만료 임박 (7일 전) | O | O | - |
| 채널 | 토큰 만료됨 (재연동 필요) | O | O | O |
| AI | 크레딧 20% 이하 | O | O | - |
| AI | 크레딧 완전 소진 | O | O | O |
| 분석 | 영상 조회수 마일스톤 (1K, 10K, 100K) | O | O | O |
| 분석 | 주간 성과 리포트 준비 | O | O | - |
| 구독 | 결제일 3일 전 | - | O | - |
| 구독 | 결제 실패 | O | O | O |

**기술 구현 방안**:

```
- WebSocket (SockJS + STOMP): 실시간 브라우저 알림
- Spring Events: 내부 이벤트 기반 알림 트리거
- 이메일: Spring Mail + 템플릿 엔진
- 카카오 알림톡: 카카오 비즈메시지 API 연동
- 알림 설정: 사용자별 알림 유형/채널 온오프 설정
```

| 항목 | 내용 |
|------|------|
| **설명** | 팀 멤버에게 역할 기반 권한 부여. 관리자/편집자/뷰어 역할 |
| **구현 난이도** | M (2주) |
| **비즈니스 임팩트** | ★★★ |
| **우선순위** | P0 |
| **예상 기간** | 2주 |

**역할 체계**:

| 역할 | 업로드 | 게시 | 예약 | 분석 열람 | AI 사용 | 설정 변경 | 멤버 관리 | 결제 |
|------|--------|------|------|---------|--------|---------|---------|------|
| 소유자 (Owner) | O | O | O | O | O | O | O | O |
| 관리자 (Admin) | O | O | O | O | O | O | O | - |
| 편집자 (Editor) | O | 승인 후 | O | O | O | - | - | - |
| 뷰어 (Viewer) | - | - | - | O | - | - | - | - |

#### 3.4.2 게시 승인 워크플로우

| 항목 | 내용 |
|------|------|
| **설명** | 편집자가 작성한 게시를 관리자/소유자가 승인 후 게시. 승인 요청 → 검토 → 승인/반려 |
| **구현 난이도** | M (2주) |
| **비즈니스 임팩트** | ★★☆ |
| **우선순위** | P1 |
| **예상 기간** | 2주 |

**워크플로우**:

```
편집자: 영상+메타데이터 작성 → [승인 요청] 클릭
    ↓
관리자/소유자: 알림 수신 → 내용 검토
```

### 4.1 AI 기반 영상 리사이징 및 편집

Publer에 없는 기능. 크리에이터의 가장 큰 반복 작업인 "플랫폼별 영상 비율 변환"을 AI로 자동화한다.

#### 4.1.1 스마트 비율 자동 변환

| 항목 | 내용 |
|------|------|
| **설명** | 16:9 원본 영상을 9:16(TikTok/Reels/Clip), 1:1(인스타 피드), 4:5(인스타 세로) 등으로 AI가 피사체 추적하며 자동 크롭 |
| **구현 난이도** | L (6주) |
| **비즈니스 임팩트** | ★★★ |
| **우선순위** | P0 |
| **예상 기간** | 6주 |

**상세 기능 명세**:

- AI 피사체 감지: 영상 프레임 분석으로 주요 피사체(사람 얼굴, 텍스트) 영역 자동 감지
- 스마트 크롭: 피사체가 프레임 밖으로 나가지 않도록 동적 크롭 포인트 조절
- 지원 비율: 16:9 → 9:16, 1:1, 4:5, 원본 유지
- 미리보기: 변환 전 썸네일 5프레임으로 결과 미리 확인
- 수동 조정: AI 결과가 불만족스러운 경우 크롭 포인트 수동 드래그 조절
- 서버 사이드 처리: FFmpeg + OpenCV (Docker 컨테이너에서 처리)

**기술 구현 방안**:

```
1. FFmpeg + face detection filter로 얼굴 기반 크롭
2. OpenCV 기반 피사체 추적 (경량 모델)
3. 또는 외부 API (Cloudinary Video Transform, AWS MediaConvert)
4. 결과물 S3 저장 → 플랫폼별 별도 URL 관리
5. Virtual Thread로 비동기 처리, 완료 시 WebSocket 알림
```

**AI 크레딧 소모**: 3크레딧/영상 (비율 변환당)

### 8.3 1인 개발 현실성 평가

현재 onGo는 1인 개발 체제이므로, 로드맵의 현실성을 고려한 우선순위 조정이 필요하다.

| Phase | 현실적 기간 | 조정 사항 |
|-------|-----------|---------|
| Phase 1 | 2-3개월 | P0 3건 + P1 핵심 5건에 집중. P2는 Phase 2로 이월 |
| Phase 2 | 3-4개월 | P0 4건 중 "AI 리사이징"과 "벤치마킹"을 우선. 워크스페이스는 Phase 2 후반 |
| Phase 3 | 4-6개월 | 마켓플레이스는 별도 팀원 영입 후 진행. 화이트라벨은 엔터프라이즈 고객 확보 후 |

**권장 사항**:

1. Phase 1 완료 후 프론트엔드 개발자 1명 영입 (계약직/파트타임)
2. Phase 2 완료 후 PM 또는 디자이너 1명 영입
3. Phase 3 마켓플레이스는 별도 비즈니스 담당자 필요
4. AI 관련 기능은 Spring AI + 프롬프트 엔지니어링으로 1인 개발 가능한 범위 유지
